# Supabase Truth Audit — Nexus OS v2

**Date:** 2026-06-30
**Repo commit:** e31a78d1af9da923c0595a8fc894820de14fe56c
**Live app:** https://nexusv20.netlify.app/

---

## Executive Summary

**The Nexus OS v2 frontend does NOT read live Supabase data for any admin dashboard section.** All 13 dashboard sections render static/bundled data from `src/data/` files or inline hardcoded arrays. Supabase is connected at the client library level but is never queried by any active UI component. The backend (Python scripts) has conditional Supabase write capability via the service role key, but the frontend is entirely static.

## 7. RLS Security Model

- **88 tables** total across 14 migrations
- **Zero anon/public policies** — entire security model requires `authenticated` role
- All policies gate on `nexus_is_active_admin()` function
- Service role key bypasses RLS (used only in Python scripts, never in frontend)
- No RLS weakening detected

---

## 8. Critical Findings

1. **Frontend is 100% static** — Zero live Supabase queries from any active UI component
2. **Supabase client is properly configured** — Anon key only, no service role in frontend
3. **Backend has conditional Supabase writes** — Python scripts can write when env is configured
4. **Parallel Supabase UI exists but is dead code** — `sections.tsx` and `Shell.tsx` are not rendered
5. **Ray Review approvals are localStorage-only** — No server persistence
6. **Hermes has no live Supabase access** — Honest stub returns "not available"
7. **Research engine writes local files, not Supabase** — Supabase inserts are conditional/gated
8. **Dashboard badges show static counts** — "64 cards", "50 candidates", "9 offers" are hardcoded
